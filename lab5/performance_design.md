# Performance Design

## 1. Анализ производительности

Главная точка нагрузки - это список отелей. Его смотрят все, в том числе незалогиненные пользователи, и каждый раз это полный scan таблицы в PostgreSQL. Второй горячий путь - проверка существования отеля при бронировании: booking-service синхронно дёргает inventory-service по HTTP, тот идёт в БД. Итого на каждое бронирование - лишний network round-trip плюс SQL.

`SearchUsersByName()` тоже медленный (LIKE без индекса), но он не горячий, так что пока не особо критично.

Условные целевые показатели: `GET /api/hotels` - не дольше 50 ms, `POST /api/bookings` - не дольше 200 ms, пропускная способность на чтение отелей - от 500 RPS.

---

## 2. Стратегия кеширования

Кешировать решил список отелей и отдельный отель по ID. Первое - потому что данные почти не меняются, а запросов много. Второе - чтобы убрать лишний SQL-запрос из пути бронирования.

Выбрал **Cache-Aside**: данные грузятся в кеш только при первом обращении, Redis поднимать не стал - для одного инстанса in-memory достаточно, и проще встроить прямо в `StorageComponent`.

TTL для списка поставил 5 минут - если кто-то добавит новый отель, он появится с небольшой задержкой, но это нормально. При `POST /api/hotels` кеш списка сбрасываю явно. Для отеля по ID - 10 минут без инвалидации, потому что отели после создания не редактируются (покааа что).

Реализация в `inventory-service/src/storage.cpp`

## 3. Стратегия rate limiting

Без ограничений на `POST /api/bookings` один пользователь может за секунды создать тысячи броней и положить MongoDB. Поэтому поставил rate limit именно сюда.

Из алгоритмов смотрел на Fixed Window - простой, но на границе окна можно получить двойной burst. Sliding Window Log точнее, но хранит каждый запрос в памяти. Остановился на **Token Bucket**: пользователь может забронировать несколько отелей подряд (в том числе и с каким-то уномоментным бёрстом), но потом токены восполняются плавно, без жёсткого сброса по таймеру.

Параметры: 10 токенов, пополнение 10/мин, ключ - `user_id` из JWT. При превышении возвращаем `429 Too Many Requests`.

К каждому ответу добавляются заголовки:
```
X-RateLimit-Limit: 10
X-RateLimit-Remaining: 7
X-RateLimit-Reset: 18
```

Реализовано в `booking-service/src/rate_limiter.cpp`

## 4. Метрики производительности

userver сам отдаёт в Prometheus латентность каждого handler (p50/p90/p99), коды ответов и статистику по запросам к PostgreSQL.

Дополнительно добавил кастомные метрики: попадания и промахи кеша (`inventory_cache_list_hits/misses`, `inventory_cache_hotel_hits/misses`) и счётчики rate limiter (`booking_rate_limiter_allowed/rejected`).

Hit rate считается так:
```promql
rate(inventory_cache_list_hits[5m]) /
  (rate(inventory_cache_list_hits[5m]) + rate(inventory_cache_list_misses[5m]))
```

## 5. Влияние оптимизаций

После кеширования `GET /api/hotels` из 20–50 ms превращается в < 1 ms, а нагрузка на PostgreSQL при 100 req/мин падает примерно со 100 запросов в минуту до одного раз в 5 минут. То же самое с проверкой отеля при бронировании.

Rate limiting при этом почти не влияет на обычного пользователя - overhead меньше 0.1 ms на мьютекс. Зато злоумышленник вместо тысяч броней за минуту отправит только 10.
